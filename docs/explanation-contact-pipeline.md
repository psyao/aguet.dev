# Explanation: the contact message pipeline

The homepage contact form, its anti-spam gates, the owner notification, and the
Infomaniak cron trigger that backstops it all look like separate features from the
outside. They're one system, built around a single constraint: **this app runs on
shared hosting with no queue worker and no reliable CLI cron.** Every design decision
below is a consequence of that constraint.

## The problem

Infomaniak shared hosting gives you PHP-as-a-web-request and a pseudo-cron
("Planificateur de tâches") that hits a URL on a 15-minute floor — it does not run
`php artisan schedule:run` on a real crontab, and there is no long-running
`queue:work` process. Two things a typical Laravel app takes for granted don't exist
here:

1. **No background delivery.** A `ShouldQueue` notification would sit in the
   `database` queue table forever with nothing to drain it.
2. **No durable scheduler.** `routes/console.php`'s
   `Schedule::command('contact:notify')->everyFifteenMinutes()` is a real schedule
   definition, but nothing on this host ever calls `schedule:run` to evaluate it.

Losing a contact-form lead because an email send timed out, or because "the cron
never actually ran," is the failure mode this whole pipeline is built to avoid.

## The approach

### Two independent delivery rails

`App\Services\ContactMessageNotifier` delivers each `ContactMessage` on two
rails — email (`notified_at` / `notify_attempts`) and kChat
(`kchat_notified_at` / `kchat_notify_attempts`) — each with its own success flag and
its own attempt counter, capped at `MAX_ATTEMPTS = 5`. A kChat webhook outage never
blocks or duplicates the email, and vice versa. Delivery is *at-least-once*: a rail
sends, then persists its flag — so a crash between send and save can retry (and, in
the email case, double-send) rather than silently lose the notification.

```
ContactForm::submit()                    contact:notify (15-min sweep)
        │                                          │
        ▼                                          ▼
  ContactMessage::create()  ──────┐        query pending rows (≤50/run)
        │                        │                 │
        ▼                        ▼                 ▼
  defer(fn () => $notifier->deliver($message))  same $notifier->deliver()
        │
        ▼
  ContactMessageNotifier::deliver()
   ├─ email pending? → Mail::send() → notified_at
   └─ kchat pending? → Notification::route('kchat', …) → kchat_notified_at
```

Both callers — the Livewire component's `defer()` closure and the artisan
command's sweep — funnel through the same `deliver()` method, so there's exactly one
place that decides whether a rail is "pending."

### DB-first, fire-and-forget fast path, durable backstop

`ContactForm::submit()` writes the `ContactMessage` row *before* attempting any
notification. Delivery is dispatched via `defer(fn () => $notifier->deliver($message))`
— Laravel's `defer()` runs the closure after the HTTP response is flushed, in the same
PHP process (no worker to hand it to; `proc_open` is disabled on this host, so even
the queue's `background` connection isn't available). This gives visitors a near-instant
success screen without waiting on an SMTP round-trip or a webhook call.

If that deferred run never completes — the process is killed, a rail throws and hits
its attempt cap for that moment, whatever — the row is already durably saved with its
delivery flags un-set. The `contact:notify` artisan command, hit every 15 minutes via
the Infomaniak cron trigger (next section), re-queries for anything still pending and
retries it. The deferred path is an optimization for the common case; the sweep is the
actual delivery guarantee.

### The Infomaniak HTTP cron trigger

Because `schedule:run` never fires here, `routes/console.php`'s schedule entry is
inert in production — it's left in place as documentation of intent, not as a working
mechanism. The actual trigger is a dedicated route:

```
GET /cron/{token}   (token: 48 hex chars)  →  CronController::__invoke()
```

registered in `bootstrap/app.php`'s `then:` callback — deliberately **not** in
`routes/web.php` — so it never enters Laravel's `web` middleware group (no session,
no CSRF, nothing that would let a HEAD request or a stray cookie leak information).
The controller:

1. Rejects `HEAD` requests outright (Laravel auto-registers `HEAD` on every `GET`
   route; a prefetcher or link-preview bot could otherwise trigger a real run since
   the token lives in a URL).
2. Validates the token with `hash_equals()` against `config('aguet.cron.token')`
   (sourced from `CRON_TOKEN` in `.env`, generated by `php artisan cron:token`),
   404-ing on any mismatch, wrong type, or missing config — never a distinguishable
   403, so there's no oracle for guessing a valid token.
3. Acquires `Cache::lock('cron:contact-notify', 240)` (non-blocking) so two
   overlapping ticks (or a retried request) can't run the sweep concurrently against
   the same rows. 240s comfortably outlives a legitimate run (≤50 rows, bounded by
   `MAX_PER_RUN`) while leaving 3-4× margin before the next 15-minute tick.
4. Calls `Artisan::call('contact:notify')` **directly** — not `schedule:run`, which
   would execute every scheduled task (handing an HTTP token the blast radius of the
   entire schedule, not just this one command) and swallows task exceptions
   internally, making its exit code meaningless.
5. Writes a `cron:contact-notify:last-success` cache key on success. This is the
   only "is the backstop alive" signal — `NotifyContactMessages::handle()` always
   returns `SUCCESS` (per-row failures are tracked in the DB, not surfaced as a
   process exit code), so `OK`/`FAILED` in the HTTP response means "did the sweep run
   at all," not "did every message send."
6. Every response the controller builds carries `Cache-Control: no-store`, and the
   whole body runs inside `try`/`catch`/`finally` so a thrown exception (from the
   lock, the cache write, or the command itself) still produces a `FAILED` (500)
   response instead of leaking to Laravel's default error page — and the lock release
   itself is wrapped separately, so a release failure can't discard an
   already-decided response.

The full decision log — four rounds of adversarial review, including two boot-time
bugs caught only by reading vendor source — lives in
[`docs/superpowers/specs/2026-07-08-infomaniak-cron-trigger-design.md`](superpowers/specs/2026-07-08-infomaniak-cron-trigger-design.md).
Read it if you're touching `CronController` or the lock/token logic — most of the
"why not the obvious simpler version" answers are there, verbatim, with file:line
citations against Laravel's own source.

### Anti-spam, entirely client-side of the DB write

`ContactForm` filters spam with three gates before a row is ever validated against
real rules — no external service, no CAPTCHA:

- **Honeypot** (`website` field) — hidden from real users via CSS, irresistible to a
  bot that fills every field it can see in the DOM.
- **Minimum fill time** — `now()->timestamp - $startedAt < MIN_FILL_SECONDS` (2s)
  catches a submit that couldn't possibly have been typed by a human.
- **Per-IP rate limit** — keyed by `ContactForm::rateLimitKey()`, which collapses an
  IPv6 address to its `/64` prefix (one ISP allocation, via `inet_pton` to correctly
  normalize compressed `::` forms) so rotating within an allocated block doesn't buy
  a fresh limit; IPv4 keys on the full address. This exists because the route-level
  `throttle` middleware doesn't apply — Livewire actions POST to `/livewire/update`,
  not the page route.

A submission that trips the honeypot or the fill-time gate is **silently accepted**:
the visitor sees the same success state as a real submission, and nothing is
validated or stored. This denies a bot any signal about which gate it hit, or whether
the gate exists at all.

### Untrusted content never reaches Markdown unescaped

The kChat notification (`App\Notifications\Channels\KChatMessage`) renders visitor-
supplied text (subject, message body) inside a Markdown card. `KChatMessage::neutralize()`
de-fangs that text before it's placed in the card — breaking leading `@mentions` and
escaping Markdown control characters — so a crafted contact-form submission can't ping
an unrelated channel member or forge formatting in the notification. The one piece of
trusted Markdown in the card is the deep link back into
`ContactMessageResource::getUrl('index', panel: 'admin')`, built server-side, never
from visitor input. `KChatChannel` also validates the webhook URL's scheme (http/https
only) and redacts the URL itself from any exception message it might log.

## Trade-offs

- **At-least-once, not exactly-once, delivery.** A crash between a successful
  `Mail::send()` and the `notified_at` save would cause a retry to send the email
  twice. Accepted because a duplicate notification email is a minor annoyance; a lost
  lead is not.
- **Silent spam rejection over an explicit error.** A false positive (a genuinely
  fast, honeypot-untouched human) sees success but nothing was sent. Accepted because
  the two gates it can trip are both hard for a real human to hit by accident, and
  showing "you were flagged as spam" would tell an actual bot exactly what to fix.
- **`OK`/`FAILED` on the cron route answers "did the sweep run," not "did every
  message send."** Per-message failures are tracked in the database instead
  (`notify_attempts`/`kchat_notify_attempts`), which the design doc treats as a
  *better* signal than a boolean HTTP status would have been — but it does mean there's
  no alerting on a stuck message today (see "Out of scope" in the design doc).
- **Token lives in a URL.** Infomaniak's pseudo-cron is GET-only; there's no way to
  put the token in a header instead. Mitigated by 404-only failure responses (no
  oracle), `no-store` everywhere, and one-command rotation
  (see [How to rotate the cron token](howto-rotate-cron-token.md)) — not eliminated.
